# H 1.0 Pro 集成说明（阶段一结论）

## 一、项目目标

H 1.0 Pro 是一个基于 **Telegram Mini App** 的高端 AI 智能交易终端。前端运行于 Telegram WebView，Bot 负责拉起 Mini App，服务器端仅负责策略信号与仿真判定，用户端负责钱包连接、本地私钥派生、签名与执行确认。

## 二、已锁定的业务参数

| 项目 | 已确认内容 |
| --- | --- |
| 首版执行层级 | **B：完整架构 + 仿真执行** |
| 利润分成比例 | **10%** |
| 归集方式 | 统一归集到 **EVM 地址**，以 **USDT** 结算 |
| 开发者归集地址 | `0x463b41d75e1018ba7a0a62f421219558ee13a7b4` |
| 主要链 | **Arbitrum / Base / Solana** |
| 资产范围 | **BTC, ETH, SOL, USDT, USDC** |
| 恒定套利 | 限于 **DEX 现货主流交易对**，以 Uniswap V3 / OKX DEX 为主 |
| 动能捕获 | 允许 **现货 + 永续合约** 混合执行 |
| 全域归集 | 允许跨链桥接；价差 > **0.8%** 且覆盖 Gas 成本才触发 |

## 四、首版系统边界

### 4.1 前端（Mini App）

前端必须承载以下职责：

1. 检测 Telegram Mini App 环境并完成初始化。
2. 渲染黑金极简交易终端界面。
3. 提供四个核心入口：
   - 实时执行记录
   - 盈亏收益日历
   - 开启智能交易
   - 分享给你的好友
4. 点击“开启智能交易”时执行双轨逻辑：
   - **分支 A：检测到 OKX 钱包** → 走连接授权流程。
   - **分支 B：未检测到钱包** → 本地派生随机私钥与地址。
5. 本地加密保存敏感材料；禁止向服务端上传私钥。
6. 展示仿真执行结果、净利润、佣金拆分与归集目标。

### 4.2 Bot / 信号端

Bot 与信号端在首版仅承担以下职责：

1. 生成 Telegram Web App 入口。
2. 下发策略状态、行情摘要、三引擎模拟指令流。
3. 生成标准化“订单意图”结构，供前端执行层消费。
4. 在订单利润结算结构中附带 **x402 分账参数占位**。
5. 默认工作在 **Testnet / 仿真模式**，但分账结构必须与正式版一致。

## 五、x402 分账结构（首版约束）

首版不是全量实盘，但必须让“抽水管”结构真实可校验。

因此需要满足：

1. 每一笔模拟获利订单都生成标准化结算对象。
2. 结算对象中必须包含：
   - 原始利润金额
   - 10% 佣金金额
   - 用户净利润金额
   - 归集链
   - 归集资产（USDT）
   - 归集地址
   - x402 分账参数对象/挂载位
3. 前端与 Bot 的界面展示必须统一以“已扣除佣金后的净利润”展示。

建议的结算模型：

```ts
interface SettlementBreakdown {
  grossProfit: number;
  commissionRate: number; // 0.1
  commissionAmount: number;
  netProfit: number;
  settleAsset: "USDT";
  settleChain: "EVM";
  collectorAddress: string;
  x402: {
    enabled: boolean;
    routeHint?: string;
    metadata?: Record<string, string>;
  };
}
```

## 六、钱包与安全边界

### 6.1 已检测到钱包

如果检测到 OKX 钱包，则前端发起钱包授权连接，并建立交易授权上下文。该流程必须尽可能减少用户操作次数，但保留清晰的连接状态、链信息与账户展示。

### 6.2 未检测到钱包

如果未检测到钱包，则前端本地派生随机私钥，3 秒内生成可用地址，做到“即开即用”。

### 6.3 安全要求

> 私钥只允许存在于浏览器本地的加密存储空间，不允许上传到任何服务端。

安全实施要求：

1. 真实私钥、助记词不进入服务端。
2. 不在代码仓库中写入任何真实凭据。
3. 所有敏感配置通过环境变量注入。
4. 仓库仅提供 `.env.example` 模板。
5. 用户消息中给出的真实密钥应视为待轮换敏感信息，不再回显到代码、日志或文档中。

## 七、界面设计哲学

已选定的视觉方向为：**瑞士国际主义 × 奢侈金融终端**。

执行原则如下：

1. 纯黑背景、白字、金色高优先级强调。
2. 避免通用落地页式居中布局，采用交易终端式分层信息结构。
3. 标题、正文、数字各自使用不同字体体系，强化秩序感。
4. 动效要像系统回馈，而不是消费级娱乐产品。
5. 严禁使用表情图标。

下一阶段优先落地前端静态骨架与交互状态机，包括：

1. Telegram WebApp 初始化上下文。

## 九、补充核验：OKX 官方文档新增结论（2026-03-27）

### 9.1 Agent Trade Kit 文档结论

根据用户提供的 OKX Agent 文档，可以进一步确认以下事实：

1. **Agent Trade Kit 以本地进程方式运行在设备上**，官方明确强调 API Key 不离开本地。
2. **官方推荐优先从模拟盘开始**，配置中支持 `demo = true` 的仿真环境。
3. **Profile 支持 `site` 字段**，可区分 `global`、`eea`、`us` 站点。
4. **Profile 支持 `proxy_url` 字段**，后续部署模板应预留代理配置位。
5. 官方工具链同时覆盖 **CLI / MCP / Skills** 形态，说明本项目首版采用“Bot 调度 + Signal 输出 + 客户端本地执行”的结构是合理的。

### 9.2 OnchainOS 文档结论

根据用户提供的 OnchainOS 官方文档，可以进一步确认以下集成边界：

1. **OnchainOS 是面向 AI Agent 的 Web3 基础设施**，并以 Skills 与 Open API 两条路径对外提供能力。
2. **Agentic Wallet** 的官方定位是：私钥生成与签名在可信执行环境（TEE）内完成，适合 Agent 原生钱包与设备侧执行场景。
3. **交易能力** 由 OKX DEX 聚合引擎驱动，原生覆盖主流 EVM 链与 Solana，符合本项目三链边界。
4. **行情能力** 可提供多链实时市场数据，适合作为三大引擎的信号输入来源。
5. **支付能力** 基于 **x402 协议**，与本项目“盈利结算瞬间挂载 10% 分账参数”的商业模型一致。

### 9.3 对 H 1.0 Pro 代码结构的直接影响

| 模块 | 直接落地结论 |
| --- | --- |
| Mini App 前端 | 继续坚持本地执行与本地私钥/会话保护，不让服务端接触用户密钥 |
| 钱包逻辑 | 首版先实现本地加密金库与 OKX 钱包检测；未来可进一步对接 Agentic Wallet / TEE 方案 |
| Signal 端 | 继续输出三链、多资产、多引擎信号快照，作为前端执行层输入 |
| x402 结算 | 结算模型必须保留 `protocol=x402`、归集地址、结算链与 metadata 字段 |
| 环境变量模板 | 必须包含 `OKX_ENV`、`OKX_SITE`、`OKX_PROXY_URL`、`COLLECTOR_EVM_ADDRESS` 等字段 |

### 9.4 当前阶段结论

截至目前，H 1.0 Pro 的首版实现路线已经具备较清晰的官方依据：

服务端负责 **信号生成、上下文调度与仿真执行快照**；Telegram Mini App 前端负责 **钱包检测、本地地址派生、用户侧执行确认与收益展示**；分账层通过 **x402 协议结构化挂载**，并统一以 **EVM / USDT** 归集到指定地址。
